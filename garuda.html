/ The hard parts notes.md

Where I had to invent something.

Four problems that didn't have a clean library answer. How I solved each.

01
Binary protocols
decode.ts

Decoding 1,500 binary packets/sec without allocating garbage.

DhanHQ's WebSocket sends three packet shapes — NSE depth (332B), NSE feed (163B), MCX feed (163B). Most JS libraries on top of these allocate a fresh object per packet. At 1,500 packets/sec, that's a GC pause every few seconds.

I wrote pure-TypeScript DataView decoders that read offsets directly and emit into pre-allocated typed events. No new per packet. No JSON.parse. The decoder routes by the first byte and slices fields straight into the demand/supply engine.

Outcome Steady-state <20% CPU on one core, <500 MB RAM for the entire monolith. Zero GC-related latency spikes.
02
Ring buffer
ring.ts

A 9 MB in-memory window across 50 symbols and 5 resolutions.

The proprietary metric — Aggressive Demand % — is a 1-hour rolling average across five resolutions (1s · 1m · 5m · 15m · 1h). Reading these from the DB on every live frame is a non-starter.

I keep one Float64Array(3600) per symbol and write to it as a circular ring. Reads at all five resolutions are derived from the same ring with one pass and a few accumulators. Total memory across 50 symbols: ~9 MB. Reads: sub-millisecond. The DB is only touched on the 1-Hz batch flush.

// ring.ts — illustrative shape class SymbolRing { buf = new Float64Array(3600); // 1-hour window head = 0; push(v: number) { this.buf[this.head] = v; this.head = (this.head + 1) % 3600; } avgOver(secs: number): number { /* O(secs), sub-ms */ } }
03
Reconnects
stream.ts

Stateless reconnects that don't leave visible gaps in the chart.

The Angular client subscribes via webSocket() with automatic retry. On every subscribe frame the server re-runs the full time_bucket query, so history and live always merge cleanly by overlapping bucket key. No server-side per-client state. No replay log to keep in sync.

Outcome Refresh the browser mid-session. Drop the network. Plug back in. The TradingView chart stitches itself back together with no visible gap.
04
Survival
wal.ts

Zero data loss when Postgres briefly disappears.

A 3-strike WAL fallback dumps any failed flush buffer to ./wal/<stream>-<ts>.json. The ring buffer keeps accepting live ticks from DhanHQ while the DB recovers — so live continuity is unaffected and historical data is recoverable on the next successful flush.

Auth is the same shape of paranoia: Argon2id password hashing, Ed25519 JWT (15-min access + same-day-capped refresh), OTP-to-admin gate, HttpOnly refresh cookies, and an in-memory refreshBlocklist that invalidates all sessions on replay-attack detection.
